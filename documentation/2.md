# Compolation Error fix

При компиляции возможна следующая ошибка, если использовать какие-то функции с макросом LIB61850_INTERNAL

```shell
[ 39%] Linking CXX executable open_server
/usr/bin/ld: CMakeFiles/open_server.dir/open_server.c.o: in function `fileAccessHandler':
/home/artm1904/Program/deception/substation/iec61850_vIED_honeypot/open_server.c:146:(.text+0x70b): undefined reference to `MmsServerConnection_getClientAddress'
/usr/bin/ld: open_server: hidden symbol `MmsServerConnection_getClientAddress' isn't defined
/usr/bin/ld: final link failed: bad value
collect2: error: ld returned 1 exit status
```

Чтобы успешно собрать проект и обойти ошибку undefined reference to 'MmsServerConnection_getClientAddress', не меняя исходный код самой библиотеки (LIB61850_INTERNAL), нужно изменить способ линковки в CMakeLists.txt.

1. В чем была причина ошибки?
В заголовочном файле функции MmsServerConnection_getClientAddress используется макрос LIB61850_INTERNAL. В Linux/GCC для разделяемых библиотек (динамических .so) этот макрос разворачивается в __attribute__((visibility("hidden"))). Это означает, что при сборке динамической библиотеки libiec61850.so символ помечается как «скрытый» — он доступен внутри самой библиотеки, но не экспортируется наружу. Из-за этого линкер выдавал ошибку: open_server: hidden symbol ... isn't defined, так как программа пыталась вызвать функцию из .so, которая спрятана.

2. Как можно обойти без изменения API? (Статическая линковка)

- Вместо того чтобы привязывать основной исполняемый файл `open_server` к динамической библиотеке `libiec61850.so`, мы заставили его линковаться статически с архивом `libiec61850.a`.
- Особенность статической линковки в том, что все объектные файлы из `.a` просто «вшиваются» в итоговый бинарник `open_server`. Так как код функции теперь находится внутри нашего же бинарника, модификатор видимости hidden больше не мешает — скрытые символы отлично видны между файлами внутри одного исполняемого файла.

1. Как это реализовано в CMakeLists.txt

В сборочный скрипт были внесены следующие изменения:

- Добавлено явное указание на статические библиотеки: Написаны дополнительные инструкции `find_library` для явного поиска файла libiec61850.a (сохранен в переменную IEC61850_STATIC_LIBRARY) и библиотеки `libhal.a` (IEC61850_HAL_LIBRARY). Статическая сборка `libiec` требует отдельного подключения слоя абстракции оборудования (HAL), так как в отличие от .so, статический архив автоматически свои зависимости не подтягивает.

- Изменена линковка только для `open_server`: В директиве `target_link_libraries(open_server ...)` мы заменили динамическую `.so` библиотеку на статические `.a`.

```cmake
target_link_libraries(open_server
    ${IEC61850_STATIC_LIBRARY}
    ${IEC61850_HAL_LIBRARY}
    pthread
    # ...
)
```

- Плагины остались нетронутыми: Плагины в папке plugin_src (например, python, ui_connector и т.д.) по-прежнему линкуются с динамической `libiec61850.so`. Эту логику сохранили, чтобы для плагинов не требовалось пересобирать саму библиотеку libiec61850 с флагом позиционно-независимого кода (-fPIC), что вызвало бы новые ошибки.

```shell
[ 21%] Built target local_sample_simulation
/usr/bin/ld: /home/artm1904/Program/deception/substation/iec61850_vIED_honeypot/third_party/libiec61850/build/stage/usr/local/lib/libiec61850.a(constr_TYPE.c.o): warning: relocation against `stdout@@GLIBC_2.2.5' in read-only section `.text'
/usr/bin/ld: /home/artm1904/Program/deception/substation/iec61850_vIED_honeypot/third_party/libiec61850/build/stage/usr/local/lib/libiec61850.a(Unsigned32.c.o): relocation R_X86_64_PC32 against symbol `asn_DEF_INTEGER' can not be used when making a shared object; recompile with -fPIC
/usr/bin/ld: final link failed: bad value
collect2: error: ld returned 1 exit status
# ...
```

В результате получили рабочий `open_server`, который использует скрытую функцию libiec61850, полностью сохранив архитектуру исходников сторонней библиотеки libiec61850.
