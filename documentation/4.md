# Разработка тест-кейсов для валидации ловушки vIED

## 1. Анализ и планирование

Сейчас проект эмулирует физику и протоколы (MMS, GOOSE, SV), а так же логирование (обнаружение атак из списка А1-А46).

## 2. Описание задачи

В коде реализовали некоторые механизмы детектирования ряда событий из перечня A1-A46. Теперь нужно верифицировать работу системы: убедиться, что при проведении атаки в лог-файл записывается корректное событие.
Написать набор команд (Bash/Python/Scapy) для имитации атак, которые позволят «триггернуть» детектор в ловушке.

Требования к тестам:
    1) Категории событий:
        - MMS (A5 - А12): Команды на чтение/запись через libiec61850 или python-mms.
        - GOOSE/SV (А1, A2, A23-A24): Скрипты на Scapy для отправки некорректных пакетов (манипуляция stNum, sqNum или инъекция в шину процесса).
        - Network/DoS (DoS Мониторинг (A31-A38): Использование hping3 или nmap для имитации SYN-флуда или сканирования портов.

    2) Структура каждого теста:
        Описание: Какое событие из списка A1-A46 мы проверяем.
        Инструмент: Какое ПО использовать (Scapy, nmap, iec61850_client).
        Команда/Скрипт: Готовый к запуску код.
        Ожидаемый лог: Пример JSON-строки, которая должна появиться в файле логов после теста.

    3) Автоматизация: Запуск этих проверки одной командой (например, через Bash-скрипт или Python-раннер), находясь внутри тестового контейнера в сети контейнеров vIED.

Особенности среды:
    Ловушка слушает стандартные порты (102 для MMS, Ethertype 0x88b8 для GOOSE).
    Логирование должно быть атомарным (одна атака = одна запись в JSON)

## 3. Разработка механизма автоматизированного запуска КА на vIED

Разработан автоматизированный Python-скрипт для симуляции всех необходимых атак (MMS Unauthorized Write, GOOSE Replay & Spoofing, GOOSE Injection Flood, MMS/TCP DoS).

Скрипт сохранен в проекте по пути: tests/simulate_attacks.py

Настройка скрипта:
Откройте файл tests/simulate_attacks.py и убедитесь, что:

TARGET_IP = "127.0.0.1" заменен на актуальный IP адрес контейнера vIED.
IFACE = "eth0" соответствует сетевому интерфейсу, который смотрит в сеть substation_default

## 4. Автоматизированный запуск и верификация

Подготовлен тестовый контейнер attacker, который теперь интегрирован в Docker-сеть subnetwork1.

Что было сделано:

1) Создан Dockerfile.attacker с установленными зависимостями (scapy, hping3, tcpdump, python3).
2) В substation.simulator.yml добавлен сервис attacker со статическим IP 10.0.0.66. Папка tests примонтирована напрямую как volume, чтобы можно было на лету редактировать Python-скрипты проверки без пересборки контейнера.
3) В скрипте tests/simulate_attacks.py целевой IP изменен на 10.0.0.2 (IP контейнера IED1_XCBR).

- Запуск всех контейнеров

```
sudo docker-compose -f substation.yml -f substation.simulator.yml up --build 
```

- Запуск скрипта для автоматизированного проведения КА

```
sudo docker exec -it iec61850_vied_honeypot-attacker-1 python3 /tests/simulate_attacks.py
```

- Проверка логов

```
sudo docker exec -it iec61850_vied_honeypot-ied1_xcbr-1 cat /var/log/tpot/vied_events.json
```

## 5. Особенности реализации

1) libiec61850 игнорирует вручную созданные GOOSE пакеты: библиотека строго проверяет, чтобы gocbRef, datSet и goID в пакете полностью соответствовали настройкам (из IED1_XCBR.ext)
2) GOOSE: Скрипт тестирования не пытается собрать GOOSE с нуля, а 15 секунд слушает сеть (scapy.sniff), ловит реальный валидный GOOSE-пакет, который публикует IED2_PTOC, и использует его как шаблон для инъекций и Replay-атак! Это 100% гарантирует, что пакет пройдет все проверки libiec61850.
3) MMS Write: Так как автоматизировать WebSockets из сырого Python-скрипта без зависимостей тяжело, скрипт не пытается делать HTTP запрос, а было написано отдельное приложение, которое можно запустить в браузере, чтобы  выполнить изменение параметра вручную через графический Web-интерфейс клиента (ведь IP клиента — 10.0.0.11, это всё равно вызовет UNAUTHORIZED_ATTACK). Например, выберите любую уставку IED1_XCBR (например, Mod.stVal) и отправьте новое значение, чтобы сэмулировать A12.
4) в чем была проблема с GOOSE Injection и DoS:

- Почему не фиксировался GOOSE Injection (Flood A23)? Логировалось кучу событий REPLAY_ATTACK_A24, но не А23. Проблема заключалась в Python-скрипте в функции update_goose. В прошлой версии скрипта было жестко задано new_stNum и инкрементировался sqNum. В итоге получалось, что каждый раз stNum в сгенерированном пакете отличался от того stNum_cache, который IED1 запомнил после легитимного пакета (от шаблонного GOOSE-пакета). Раз stNum не совпадал, C++ расценивал это не как флуд внутри одного состояния (delta < 10 && current_stNum == inputVal->stNum_cache), а просто как странный старый пакет с чужим stNum и ругался на Replay (9 < 16)
- Решение: Переписана функция генерации пакета. Теперь она называется update_goose_sqnum и вообще не трогает stNum. Она берет оригинальный пакет-шаблон, сохраняет его stNum и просто быстро спамит sqNum=(i+1). Это 100% должно затриггерить проверку current_stNum == inputVal->stNum_cache и, как следствие, INJECTION_A24: Unrealistic packet frequency

## 6. С какими ограничениями я столкулся при реализации автоматизированного скрипта

1) DoS: В виртуальной Docker-bridge сети потоки Python не успевали генерировать > 500 TCP-подключений в секунду или > 3000 GOOSE-пакетов из-за накладных расходов ОС, поэтому счетчик mms_req и goose_req не достигал порогов для триггера события. Пришлось снизить пороги детектирования DoS в C++ коде (open_server.c) до 20/sec для MMS и 30/sec для GOOSE.
2) GOOSE Injection: Сложно детектировать в докер-контейнерах, потому что скрипт генерировал пакеты с Delta < 5ms, но функция sendp в Scapy работает медленно, и реальная разница во времени (измеряемая в миллисекундах Hal_getTimeInMs() в C++) получалась больше 5 мс. Это особенность таймингов.
