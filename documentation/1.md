# HoneypotLogger Implementation

## 1. Анализ и планирование

Сейчас проект эмулирует физику и протоколы (MMS, GOOSE, SV). Мне нужно реализовать модуль Security Logger, который будет записывать действия атакующего в JSON-файл для последующего парсинга через Logstash в составе T-Pot.

## 2. Описание задачи

1. Спроектировать и реализовать класс на C++ (например, HoneypotLogger), который будет отвечать за структурированное логирование. Design JSON Schema for T-Pot compatibility
2. Требования к логам:

```shell
    Формат: Строго JSON (одна строка — один объект).
    События для отслеживания:
        MMS: Попытки чтения/записи переменных (Data Objects), команды управления (Select Before Operate), аутентификация.
        GOOSE: Детектирование входящих сообщений (не со своего ли узла они пришли?), попытки инъекции.
        Системные: Попытки доступа к файлам через эмулируемый веб-интерфейс или SSH.
```

1. Поля в JSON:

```shell
        timestamp (ISO8601)
        src_ip / src_port
        protocol (MMS, GOOSE, WEB, SSH)
        action (READ, WRITE, COMMAND, LOGIN)
        target (например, имя логического узла MMXU1.TotW.mag.f)
        value (какое значение пытались записать)
        status (SUCCESS, DENIED, ERROR)
```

## 3. Архитектура интеграции Security Logger для vIED

1. Архитектура: Точки перехвата событий в iec61850_open_server
Поскольку сервер open_server.c основан на C-библиотеке libiec61850, логгер на C++ должен экспортировать C-совместимый интерфейс. Вызовы логгера следует встроить в следующие ключевые точки:

`MMS (Чтение, Запись, Команды, Доступ к файлам)`

- Файловый доступ (open_server.c): Функция fileAccessHandler. Сюда добавляем хук HoneypotLogger_LogFileAccess(...).
- Чтение/Запись переменных: Необходимо зарегистрировать обработчики для MMS подключений: IedConnection_installReadAccessHandler и IedConnection_installWriteAccessHandler. Они вызываются при каждой попытке клиента прочитать или записать DataSet/DataAttribute. Внутри этих хендлеров мы логируем действие, запрашиваемый узел (например, MMXU1.TotW.mag.f) и значение, после чего возвращаем разрешающий или запрещающий код доступа (для правдоподобности ханипота можно позволить "записывать" только определенные переменные, а попытки изменения остальных логировать как инцидент).
- Команды управления (Select Before Operate / Direct Operate): Использовать IedConnection_installControlHandler. Здесь мы логируем попытки выполнения команд, например, отключения выключателя (XCBR1.Pos.Oper).

`GOOSE (Инъекции и Аномалии)`

- Штатный прием: В inputs.c, функция subscriber_callback_inputs_GOOSE. Здесь извлекается GooseSubscriber_getSqNum и GooseSubscriber_getAppId.
- Детектирование спуфинга: Чтобы детектировать чужие GOOSE-пакеты, можно добавить проверку Source MAC Address (или stNum / sqNum). Если приходит GOOSE с MAC-адресом, отличным от ожидаемых сетевых маршрутов, или с невалидными номерами последовательности (например, атака GOOSE replay или inject), логгер отправляет событие: action="INJECTION_ATTEMPT".

## 4. Что было сделано

1. Создан С++ Класс (utils/honeypot_logger.hpp):

- Реализована безопасная поточная (атомарная) запись в JSON-файл (по умолчанию /var/log/tpot/vied_events.json) на базе библиотеки nlohmann::json.

```shell
mkdir -p inc/nlohmann && wget -O inc/nlohmann/json.hpp https://github.com/nlohmann/json/releases/download/v3.11.2/json.hpp
```

- Поддерживает генерацию таймстемпов в формате ISO8601.

1. Создан C-API интерфейс (inc/honeypot_logger_c_api.h & utils/honeypot_logger_c_api.cpp):

- Чтобы использовать C++ логгер в C-базовом приложении сервера, написаны функции-обертки Logger_LogFileAccess, Logger_LogGooseAnomaly, Logger_LogMmsAction.

1. Обновлен CMakeLists.txt:

- Язык проекта переключен на C CXX.
- Подключена библиотека stdc++ для успешной линковки модуля журналирования.

1. Интегрированы перехватчики (Hooks):

- В open_server.c: Инициализация логгера (Logger_Init()) и логирование попыток взаимодействия с файловой системой (чтение, удаление, переименование).
- В input/inputs.c: Добавлен вызов генерации события аномалии при несовпадении SqNum у пришедших GOOSE-сообщений (потенциальные инъекции).

1. Компиляция libiec61850:

- В связи с отсутствием исходников libiec61850 в прямом доступе текущего проекта (необходимых для сборки), исходный код библиотеки был склонирован в third_party/libiec61850.
- Библиотека успешно скомпилирована, и пути перенастроены. Все модули слинкованы.

```shell
git clone https://github.com/mz-automation/libiec61850.git third_party/libiec61850 && cd third_party/libiec61850 && mkdir build && cd build && cmake .. && make -j4
```

- Сервер open_server успешно собран с поддержкой нового логгера (cmake --build .).

```shell
cd third_party/libiec61850/build && make install DESTDIR=$PWD/stage && cd ../../../build && cmake -DIEC61850_ROOT=$PWD/../third_party/libiec61850/build/stage/usr/local .. && make -j4
```
